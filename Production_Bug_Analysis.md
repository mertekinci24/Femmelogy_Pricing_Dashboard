## Bug 1 — Amazon Fresh CSV Import: "Maliyet Gir" Warning & Target Margin Shows "—"

### Symptom
When uploading a **fresh Amazon Seller Central CSV report**, new products are created in the table but show `⚠️ Maliyet Gir` in the Recommended Price column and `—` in the Target Margin column. However, importing an **internal system export** (previously generated by the app's own "Export" feature) works correctly.

---

### Root Cause Analysis

The bug has **two distinct layers** that compound each other.

#### Layer 1 — Data Path Divergence (Primary Cause)

There are **two completely separate import functions** in `app.js`:

| Function | Trigger Condition | What it sets |
|---|---|---|
| `processAmazonRawCSV()` (Line 661) | Fresh Amazon CSV (headers: SKU, ASIN, Ürün Başlığı…) | `maliyet: 0, ambalaj: 0, sabit: 0, bireyselMarj: null` |
| `processSystemAmazonBulk()` (Line 2116) | Internal system export (headers: "Ürün Maliyeti (₺)", "Hedef Marj (%)"…) | Reads `parsedMaliyet`, `parsedMarj` from correct columns |

The router that decides which function to call is at line **2256–2261**:

```javascript
// Line 2256-2261 (import handler)
if (isSystemFile) {
  processSystemAmazonBulk(rows);    // ← Has cost columns → works
} else {
  processAmazonRawCSV(metin);       // ← No cost columns → maliyet = 0
}
```

`isSystemFile` is detected by checking if the header contains `"Ürün Maliyeti (₺)"` or `"Hedef Marj (%)"`. A fresh Amazon CSV never has these columns, so it always routes to `processAmazonRawCSV`.

#### Layer 2 — Hardcoded Zero COGS in `processAmazonRawCSV` (The Bug)

Inside `processAmazonRawCSV`, the branch for **new products** (line 719–732) always creates the product with `maliyet: 0`:

```javascript
// Line 720–731 — NEW product branch
const r = amazonHesap(0, 0, 0, GP.kargo, GP.marj, "kozmetik", buyboxPrice, currentPrice);
//                    ↑ ↑ ↑
//    maliyet=0, ambalaj=0, sabit=0 — ALWAYS zero, no user input possible here
if (!r.hata || r.missingCogs) {
  STATE.amazon.push({
    id: Date.now() + i,
    ad, sku, asin,
    maliyet: 0, ambalaj: 0, sabit: 0, bireyselMarj: null,
    ...
  });
}
```

Then `amazonHesap()` checks at line **144**:

```javascript
// Line 144 — amazonHesap entry guard
const baseCost = maliyet + ambalaj + sabit;
if (baseCost <= 0) {
  return {
    hata: "Maliyet Eksik",
    missingCogs: true,   // ← This flag triggers the "⚠️ Maliyet Gir" UI
    satisF: 0, ...
  };
}
```

The `missingCogs: true` flag propagates to `amazonRender()` at line **562–572**:

```javascript
// Line 562–572 — amazonRender
const isMissingCogs = p.missingCogs === true;
if (isMissingCogs) {
  recPriceHtml = `⚠️ Maliyet Gir`;    // ← "Maliyet Gir" warning shown
}
const targetMarginHtml = isMissingCogs ? '—' : `%${targetMargin}`;  // ← "—" for margin
```

**This is the correct and intended behavior** — `processAmazonRawCSV` intentionally creates zero-cost placeholder products since a fresh Amazon CSV has no cost data. The "warning" is a UX prompt telling the user to open the edit modal and enter their costs.

#### Layer 3 — Why the Internal Export Works

`processSystemAmazonBulk` (line 2125–2128) reads costs from named columns:

```javascript
const parsedMaliyet = parseExcelNum(row["Ürün Maliyeti (₺)"]) || 0;
const parsedMarj    = parseExcelNum(row["Hedef Marj (%)"]);
```

These columns exist in the internal export because the app itself generates them. So re-importing an internal export correctly populates all costs.

#### Conclusion — This is a UX Gap, Not a Code Bug

The math engine, guardrails, and rendering are all working **exactly as designed**. The root cause is that the **user doesn't understand the two-step workflow**:
1. Upload fresh CSV → products created as zero-cost placeholders
2. Click "Düzenle" on each row → enter Maliyet, Ambalaj, Sabit → Recommended Price appears

**The fix is a UX improvement**: display a clear inline tooltip or import-completion message explaining this workflow. Optionally, add a bulk-edit mode or a post-import wizard.

---

### Proposed Technical Solution

**Option A (Minimal — UX copy fix):** After `processAmazonRawCSV` completes, show an extended feedback message:

```javascript
// In gosterFeedback call at line 742
const finalMsj = eklendi > 0
  ? `${eklendi} yeni ürün eklendi. ⚠️ Önerilen fiyat görebilmek için her ürünü düzenleyerek maliyet bilgilerini giriniz.`
  : "CSV'den geçerli ürün bulunamadı.";
```

**Option B (Full — Bulk Cost Entry modal):** After CSV import, open a "Cost Entry Wizard" modal that lists all newly imported zero-cost products and allows bulk cost entry before saving to STATE.

## Bug 2 — "Mevcut Fiyat (₺)" Input Missing from Edit/Add Modals

### Symptom
When adding a new product or editing an existing one, there is no `"Mevcut Fiyat (₺)"` input field in either the Amazon or Trendyol modals. The column exists in the table view but cannot be set manually by the user.

---

### Root Cause Analysis

#### Confirmed by HTML Inspection

The Amazon modal HTML spans `index.html` lines **382–467**. It contains these input fields:

```
modal-name     ✅  Ürün Adı
modal-sku      ✅  SKU
modal-asin     ✅  ASIN
modal-cost     ✅  Ürün Maliyeti (₺)
modal-pkg      ✅  Ambalaj Maliyeti (₺)
modal-fixed    ✅  Sabit Maliyet/Birim (₺)
modal-margin   ✅  Hedef Marj (%)
modal-amz-category ✅ Kategori
```

`modal-current-price` / `modal-currentprice` — **ABSENT** ❌

The Trendyol modal spans `index.html` lines **469–550**. Similarly, no `currentPrice` input exists.

#### How `currentPrice` Is Currently Read in `app.js`

In `modalAc()` (line 358–378), when editing an existing product:

```javascript
// Line 358–367 — modalAc
if (urunId) {
  const p = STATE.amazon.find(x => x.id === urunId);
  document.getElementById("modal-name").value   = p.ad || "";
  document.getElementById("modal-cost").value   = p.maliyet || 0;
  // ... other fields
  // currentPrice is NEVER written to any input — it's read-only from STATE
}
```

In `modalKaydet()` (line 489), `currentPrice` is sourced from existing STATE data only:

```javascript
// Line 488–490 — modalKaydet
const existingProduct = modalMod ? STATE.amazon.find(x => x.id === modalMod) : null;
const currentPrice = existingProduct ? (existingProduct.currentPrice || 0) : 0;
// ↑ For NEW products (modalMod=null), existingProduct=null → currentPrice always 0
```

**For new products:** `currentPrice` is hardcoded to `0` — the user has no way to provide it manually.  
**For existing products:** `currentPrice` is only updated via CSV import (`processAmazonRawCSV` line 716) or the `parseBuyboxFiyat` parser — never via the edit modal.

This means: if a user adds a product manually (not via CSV), the "Mevcut Fiyat" column in the table always shows `—` and the "Amazon Mevcut Satış Performansı" preview block (lines 452–457) is always hidden.

---

### Proposed Technical Solution

**Amazon Modal — Add input field** in `index.html` after the existing ASIN row (around line 395):

```html
<div class="modal-row">
  <div class="modal-field">
    <label for="modal-current-price">Mevcut Fiyat (₺)
      <span class="field-hint">isteğe bağlı</span>
    </label>
    <div class="iw"><span class="ip">₺</span>
      <input type="number" id="modal-current-price" placeholder="0,00" min="0" step="0.01" />
    </div>
  </div>
</div>
```

**`app.js` — Wire up the new field in 3 places:**

```javascript
// 1. modalAc() — populate on edit
document.getElementById("modal-current-price").value = p.currentPrice || "";

// 2. modalAc() else branch — clear on new
temizle(..., "modal-current-price");

// 3. modalKaydet() — read value
const currentPrice = parseFloat(document.getElementById("modal-current-price").value) || 0;
```

**Same pattern applies to the Trendyol modal** — add `modal-ty-currentprice` and wire it to `modalTyAc()` and `modalTyKaydet()`.

## Bug 3 — Auth Screen Flash on Page Refresh

### Symptom
When an authenticated user presses Ctrl+F5 (hard refresh), the Login screen flashes for approximately **1 second** before disappearing and showing the dashboard.

---

### Root Cause Analysis

#### The Auth Overlay Starts Visible

`#auth-overlay` is defined in `index.html` at line **24** with `display:flex` hardcoded in the inline style:

```html
<!-- index.html Line 24-29 -->
<div id="auth-overlay" style="
  position:fixed;inset:0;z-index:9999;
  background:linear-gradient(...);
  display:flex;         ← VISIBLE FROM FIRST PAINT
  align-items:center;justify-content:center;
">
```

On every page load, the login card is **immediately visible** from the browser's first paint.

#### The `onAuthStateChanged` Callback Has a Network Latency

The auth resolution chain in `app.js` (appended at the end of the file) is:

```
DOMContentLoaded fires
    ↓
_bootstrapFirebaseAuth() called
    ↓
Polls for window.DB (600ms intervals until firebase-db.js module loads)
    ↓
window.DB.onAuthChange(callback) registered  ← async module boundary
    ↓
Firebase SDK makes network call to verify cached token
    ↓
onAuthStateChanged fires (~300–800ms later)
    ↓
_showDashboard() called → overlay.style.display = 'none'
```

The gap between **first paint** (overlay visible) and **`_showDashboard()` called** is typically **600–1500ms** — this is the flash.

#### Why the Polling Makes It Worse

The `_bootstrapFirebaseAuth()` function starts with a 600ms poll interval:

```javascript
function _bootstrapFirebaseAuth() {
  if (!window.DB || typeof window.DB.onAuthChange !== 'function') {
    setTimeout(_bootstrapFirebaseAuth, 600);  // ← adds 600ms minimum delay
    return;
  }
  ...
}
```

Even if `window.DB` is available immediately, the function still adds at least one 600ms retry cycle before registering `onAuthStateChanged`, creating a guaranteed minimum **600ms flash window** on top of Firebase's own latency.

---

### Proposed Technical Solution

**Three-state approach:**

| State | Overlay Display | Card Content |
|---|---|---|
| Loading (initial) | `display:flex` (blocks dashboard) | Spinner + "Yükleniyor…" |
| Authenticated | `display:none` | — |
| Signed out | `display:flex` | Login form |

#### Step 1 — Change initial overlay content to a neutral loading state

In `index.html`, replace the initial overlay content with a loading spinner. The login form is injected dynamically by JS only when `onAuthStateChanged` confirms the user is signed out:

```html
<!-- index.html — Initial auth overlay state -->
<div id="auth-overlay" style="position:fixed;inset:0;z-index:9999;
  background:linear-gradient(135deg,#0d0f14,#131720);
  display:flex;align-items:center;justify-content:center;">
  
  <!-- Loading state (shown immediately) -->
  <div id="auth-loading">
    <div style="/* F logo + spinner animation */"></div>
    <p style="color:#94a3b8;font-size:13px;margin-top:16px;">Yükleniyor…</p>
  </div>
  
  <!-- Login form (hidden initially, shown only if signed out) -->
  <div id="auth-login-card" style="display:none;">
    <!-- ... existing login form ... -->
  </div>
</div>
```

#### Step 2 — Reduce bootstrap polling interval

```javascript
// In _bootstrapFirebaseAuth() — reduce from 600ms to 100ms
setTimeout(_bootstrapFirebaseAuth, 100);
```

#### Step 3 — Show login form only after auth state is confirmed

```javascript
// In _bootstrapFirebaseAuth onAuthChange handler
window.DB.onAuthChange(function(user) {
  if (user) {
    document.getElementById('auth-overlay').style.display = 'none';
  } else {
    // Auth confirmed: user IS signed out — now show login form
    document.getElementById('auth-loading').style.display = 'none';
    document.getElementById('auth-login-card').style.display = 'block';
  }
});
```

This eliminates the flash entirely: authenticated users see the loading spinner for <300ms (Firebase token verification time) before the overlay disappears. Unauthenticated users see the spinner briefly before the login form appears.

> [!NOTE]  
> Bug 1 is **not a defect** — it is intentional behavior. The "Maliyet Gir" warning is the system's correct response to zero-cost products created from a raw Amazon CSV. The fix is UX messaging, not logic.

> [!IMPORTANT]  
> Bug 2 has a downstream impact on the **Mevcut Satış Performansı** analytics block in the modal. Even when the user edits an existing product that had its `currentPrice` set via CSV import, the edit modal does not display the current price, making it invisible and non-editable — a data loss risk if the user saves without knowing the value was there.

> [!WARNING]  
> Bug 3, while appearing cosmetic, can **erode user trust** in a production SaaS application. The login flash makes the app appear insecure or unstable on refresh, especially on slower connections where the flash duration can exceed 2 seconds.
